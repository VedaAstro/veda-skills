# VEDA Sales Playbook

## Когда использовать

- Готовишь менеджера к звонку по холодной или тёплой базе.
- Разбираешь конкретное возражение на звонке, в переписке или после диагностики.
- Проектируешь resurrection-цепочку для спящих лидов.
- Даёшь менеджеру или РОПу coaching по записи разговора.
- Нужны нормы по разговору: tempo, listen/talk, глубина discovery, next step.
- Нужно оценить готовность сделки без фантазий и харизмы "на глаз".

## Что это делает

- Даёт воспроизводимые frameworks, а не "продай через состояние".
- Переводит разговор в структуру: боль -> последствия -> решение -> следующий шаг.
- Помогает Асe выдавать рекомендации по звонкам и переписке без манипуляций.
- Держит VEDA в премиальном тоне: мягко, ясно, без false scarcity и NLP-фокусов.

## Decision Router

- IF холодный звонок или холодная база -> `reference/scripts-cold.md`
- IF тёплая заявка с лендинга, квиза, формы -> `reference/scripts-warm.md`
- IF диагностика 1-на-1, high-ticket консультация -> `reference/scripts-diagnostic.md`
- IF сопротивление, сомнение, "подумаю", цена, супруг, недоверие -> `reference/objections-handbook.md`
- IF спящий лид больше 14 дней -> `reference/resurrection-sequences.md`
- IF нужно быстро оценить зрелость сделки -> `reference/meddic-veda.md`
- IF нужен call-coaching и нормы разговора -> `reference/benchmarks-gong.md`
- IF нужны реальные social proof и кейсы -> сначала `cases-registry`, не выдумывать

## Протокол ответа для Аси

1. Определи тип контакта: `cold`, `warm`, `diagnostic`, `close`, `resurrection`.
2. Выдели цель текущего шага: встреча, диагностика, оплата, возврат в диалог.
3. Выбери framework, не смешивай всё сразу:
   - cold -> SPIN-lite + Cialdini
   - warm -> MEDDIC-lite + SPIN
   - diagnostic -> Sandler + Challenger + айкидо
   - objection -> айкидо + SPIN follow-up
   - dormant -> resurrection cadence
4. Дай пользователю:
   - краткий разбор ситуации;
   - рекомендуемый сценарий;
   - 3-5 точных фраз;
   - красные флаги;
   - следующий шаг.
5. Если просят "разбери звонок", комментируй по наблюдаемым паттернам, а не по интуиции.

## Как выбирать framework

### SPIN

- Бери первым, когда клиент ещё не разложил свою ситуацию словами.
- Работает лучше, чем pitch, когда продаётся трансформация, а не "вещь".
- Особенно полезен в VEDA, где решение часто покупают после осознания последствий.

### Challenger

- Включай, когда клиент принёс неверное объяснение своей проблемы.
- Не спорь. Сначала присоединись, затем покажи overlooked pattern.
- В VEDA Challenger-момент должен звучать как insight, а не как "я лучше знаю".

### Gong

- Применяй для коучинга менеджеров и review звонков.
- Нормы нужны как guardrails, не как дубинка.
- Если разговор живой и выигрывает сделку, не фетишизируй цифры без контекста.

### MEDDIC

- Включай, если решение не равно "нравится / не нравится".
- Особенно важно, когда есть супруг, семейный бюджет, дедлайн потока, сравнение школ.
- На B2C языке VEDA MEDDIC = "что должно совпасть, чтобы человек купил без сомнений".

### Sandler Pain Funnel

- Применяй глубже в диагностике, когда клиент уже доверяет и готов открываться.
- Не начинай звонок с дна воронки. Сначала контакт и permission.
- Цель не "расковырять боль", а помочь клиенту увидеть цену бездействия.

### Cialdini

- Используй как этический слой поверх любого скрипта.
- Лучшие принципы для VEDA: authority, social proof, commitment, unity.
- Scarcity применять только при настоящем ограничении: поток, дедлайн, личный слот.

### Resurrection

- Работает, когда лид не сказал "нет навсегда", а просто выпал из внимания.
- Основной принцип: value-first, not guilt-first.
- Всегда уважай opt-out и не дави на чувство вины.

### Айкидо

- Базовый стиль VEDA на возражениях.
- Сопротивление не ломаем. Сначала признаём, затем поворачиваем к более точному вопросу.
- Формула: "Да, это понятно" -> "Давай уточню" -> "Тогда смотрим вот сюда".

## Правила VEDA

- Айкидо > борьба.
- Внешний локус: причина в механике, периоде, паттерне, а не "с тобой что-то не так".
- Голос Ирины подключай только если пользователь просит текст от её имени или нужен тёплый closing после сформированного trust.
- Реальные кейсы бери только из `cases-registry` или `Playbook_Cases.md`; если реестр не открыт, говори "нужен кейс из реестра", а не выдумывай.
- Не называй внутренние деньги: GMV, выручка, количество оплат, внутренние чеки.
- Продукты называй категориями: `флагман`, `бюджетный`, `upsell`, если нет публичной цены на лендинге.
- Любой urgency должен быть настоящим и проверяемым.

## Что НЕ работает

- **"Продажи через миссию" как единственный механизм.**
  Красиво звучит, но без discovery, qualification и next step не даёт воспроизводимости.
- **Харизма и "3 волшебных вопроса".**
  Это не система. На команде не масштабируется и не коучится по записям.
- **Копирование Hormozi слово в слово.**
  Его DTC/fitness риторика даёт скорость, но без адаптации ломает доверие в high-trust edtech.
- **False scarcity.**
  "Осталось 3 места" без факта разрушает премиум и повышает защиту.
- **Manipulative NLP.**
  Да-сеты, подстройка-переводка ради скрытого давления и псевдогипноз снижают trust и дают токсичную послевкусие.
- **Слишком ранняя презентация продукта.**
  Пока не названы pain, stakes и fit-критерии, рассказ о программе звучит как push.
