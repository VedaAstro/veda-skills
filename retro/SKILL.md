# Ретро — root cause анализ + обеспечение применения

## Главное правило

Ретро **не описывает теорию в памяти**, а **обеспечивает применение опыта в будущем**. Декларативное правило не работает — повторяющиеся косяки тому пруф. Поэтому каждый урок закрывается **самым сильным механизмом из доступных**, не дефолтным `feedback_*.md`.

Лестница механизмов (от сильного к слабому):

| Тип root cause | Механизм |
|---|---|
| Поведенческий, повтор после feedback (декларативно не сработал) | **Hook в settings.json** — блокирует действие при нарушении |
| Класс ошибок в выводе (ссылки, айдишки, форматы массовой генерации) | **Регрессионный тест / гард** — `scripts/check_*.sh` или pre-tool-use hook на массовые операции |
| Инфра / код молча падает | **Watchdog / код-фикс / deploy guard** — техническая защита, не правило |
| Забыл прочитать MEMORY.md / Decision Router перед задачей | **Pre-flight в команде/скилле** — Read-блок в `.claude/commands/X.md` или `SKILL.md` |
| Действительно новое знание, ни разу не сталкивался | **Feedback в памяти** — последняя инстанция, не первая |

Главный вопрос по каждому уроку: **«Это знание у меня уже было?»**. Если да — feedback запрещён, эскалация на ступень выше.

## Когда использовать

- Сессия > 30 мин, архитектура, новый процесс, сложный дебаг
- Затронуты skills/правила/промпты — нужна регрессия
- Сессия «что-то странно сделалась», поймать паттерны

НЕ нужен:
- Простые баг-фиксы по понятной причине
- Чтение кода без артефактов
- Короткие правки docs/memory

## Структура: 4 фазы + финальный план

DETECT → EXECUTE → SCHEDULE → REPORT (5 блоков, последний — план действий).

---

### 1. DETECT — найти паттерны

Пройти по чек-листу ловушек. Помечать только реально случившиеся.

**Ключевые ловушки:**
- залепил симптом — фикс работает, root cause не найден/не записан
- ускорил старое вместо перестройки процесса
- метрика/дашборд без замыкающей петли (кто смотрит, что делает по результату)
- метрика без контрметрики (что сломали оптимизируя одно)
- «запустим, потом проверим» — нет регрессионных тестов на изменённое
- bus-factor = 1 (всё держится на Алексе или на мне)
- решение medium/low уверенности не зафиксировано в `memory/decisions/`
- scope creep — расширил задачу за пределы запроса
- поспешил без верификации (web_search / git log / чтение кода пропущены)
- лестница «быстро/средне/надёжно» вместо одного устойчивого варианта (`feedback_pick_strongest_option.md`)
- английские слова в русском тексте без необходимости (`feedback_no_english_in_russian.md`)
- выдал «справку» вместо действия — описание проблемы без её закрытия

**Для каждой пойманной ловушки — два вопроса:**
1. **Root cause** — почему случилось? Не «забыл», а «что в потоке работы делает забывание возможным».
2. **История повторений** — это **1-й, 2-й или 3-й+ раз**? Проверить grep по `memory/feedback_*.md` и `decisions/*.md` на похожий косяк.

DoD для код-задач (если применимо): psql claude_ro вывод / curl с JWT / preview_screenshot / git diff --stat / чек-лист скилла прогнан. Если research / docs / memory — пропустить.

---

### 2. EXECUTE — закрыть самым сильным механизмом

По каждому уроку выбрать механизм из лестницы выше **по правилу повторений**:

- **1-й раз**, новое знание → можно `feedback_*.md` + pre-flight в скилле
- **2-й раз** → feedback не сработал, эскалация: pre-flight **+** регрессионный тест/гард
- **3-й+ раз** → декларативно не лечится, нужен **hook** в settings.json или код-фикс

Действие — **сразу** в этой же сессии (Edit / Write / Codex / scheduled-task), не описание в чате.

| Тип закрытия | Что делаю |
|---|---|
| Hook | Edit `~/.claude/settings.json` (или вызов skill `update-config`) — pre-tool-use блок на нарушение |
| Регрессионный тест / гард | Codex по `platform/scripts/codex_launcher.sh` с полной спекой; путь — `scripts/check_<slug>.sh` или test-кейс в существующем suite |
| Код-фикс / watchdog / deploy guard | Edit код или Codex; обязательно — `[НЕ ЗАДЕПЛОЕНО]` пометка, деплой через план действий с ОК |
| Pre-flight в команду/скилл | Edit `.claude/commands/<cmd>.md` или соответствующий `SKILL.md` — добавить блок «Прочитай: …» |
| Feedback в памяти (fallback) | Write `memory/feedback_<slug>.md` + строка в `MEMORY.md` Decision Router |
| Decision-файл | Write `memory/decisions/YYYY-MM-DD-<slug>.md` со всеми полями (см. шаблон ниже) |
| Откат правила (вызвало регрессию) | Edit `memory/feedback_<slug>.md` — пометить `status: rolled_back`, ссылка на git-версию до отката |

**Правило версионирования:** при изменении скилла или правила — bump поля `version:` в frontmatter и комментарий в git что менялось. Без версии откат превращается в «попробуем вернуть как было».

**Шаблон файла решения** (используй при Write, не выводи в чат):

```markdown
---
date: YYYY-MM-DD
slug: short-kebab-case
process: sales | astrolog | hr | webinar | infra | skill | other
confidence: high | medium | low
review_after: YYYY-MM-DD     # high=90д, medium=30д, low=7д
status: open
version: 1
---

# <Заголовок одной строкой>

## Вход
Контекст. Данные. Ограничения. Хватит на воспроизведение через 3 месяца.

## Альтернативы
- A: ...
- B: ...

## Решение
Одно предложение.

## Обоснование
Почему именно этот вариант.

## Уверенность
high / medium / low + почему.

## Что считается успехом
Конкретный измеримый критерий с числом и сроком.

## Контрметрика
Что отслеживаем, чтобы не выиграть в одном проиграв в другом.

## Открытые задачи (закрыть до review_after)
- ...

## Результат (заполнить на review_after)
- Что произошло:
- Метрика: было / стало
- Контрметрика: было / стало
- **Повторился ли тот же паттерн?** да / нет / частично — главный вопрос review
- Что узнали (кандидат в feedback или эскалация механизма):
- Следующее решение:
```

### 3. SCHEDULE — review с проверкой повтора, не выполнения

Для каждого нового/обновлённого decision-файла ставим напоминание (без него — архив, Oper8 §2.3):

```
mcp__scheduled-tasks__create_scheduled_task(
  taskId: "review-<slug>",
  fireAt: "<review_after>T09:00:00+03:00",
  description: "Проверка повтора паттерна по <slug>",
  prompt: "1) grep memory/ и git log за период на ключевые слова паттерна.
           2) Если повторился — feedback не работает, эскалировать на ступень выше
              в лестнице механизмов (запиши новый decision с этой эскалацией).
           3) Если не повторился — заполни «Результат» через Edit, закрой статус.
           4) Промежуточный исход — продли review на 14 дней."
)
```

`review_after` — по уверенности: high → 90д, medium → 30д, low → 7д.

Дополнительно: при `/retro` после нетривиальной сессии — если в системе нет еженедельного обхода `_overdue/` decisions с алертом в Telegram при просрочке, создать его (cron `0 9 * * 1`). Алерт обязателен — иначе «метрический театр» (Oper8 §2.4).

---

### 4. REPORT — отчёт в чат

Целевое разрешение — **«среднее»**: не стабы, не простыни. Каждый блок несёт минимум контекста для решения без перечтения сессии. 5 блоков + план действий **последним**. Если блок пуст — «—».

**1. УРОКИ** (1-3 пункта). Формат:

```
1. <Косяк одной фразой> [1-й / 2-й / 3-й+ раз]. Корень: <причина>.
   → Механизм: <hook / тест / код-фикс / pre-flight / feedback>. <Путь артефакта>.
```

Если 3-й+ раз — явно отметить «декларативно не работает, эскалирую на <следующая ступень>».

**2. АРТЕФАКТЫ** (одна строка на каждый):

```
- memory/feedback_<slug>.md — <правило 5-8 слов>
- memory/decisions/<file>.md — <что обновлено/создано>
- ~/.claude/settings.json — добавлен hook <slug>
- <код-путь:строка> — <что изменено> [НЕ ЗАДЕПЛОЕНО / ЗАДЕПЛОЕНО]
- scheduled task review-<slug> на <дата>
```

**3. ЛУЧШЕЕ В СЕССИИ** — одна строка, конкретное действие.

**4. СТРАТЕГИЧЕСКИЙ ДОЛГ** — только по триггерам (та же боль 3-й раз, решение medium/low без петли, метрика без контрметрики, bus-factor=1, отложили «сначала наведём порядок»). Иначе — «—».

Если триггер есть — одна ставка по фильтрам (актив нельзя купить / разблокирует 3+ процессов / владелец не Алекс не я / дёшево стартует) + барьеры (измеримо / данные есть / организация может действовать). Не подгонять ✅✅✅ ради засчитывания.

**5. ВЕРДИКТ** — одна строка: «Закрыто X из Y. Остаток в плане ниже» или «Сессия закрыта чисто».

---

### 5. ПЛАН ДЕЙСТВИЙ — финальный блок (всегда последним)

Топ-CTO позиция: **я приоритизирую и беру ответственность за рекомендацию**. Не «выбери из меню», а «вот мой маршрут — тормози если не согласен». **Порядок списка = приоритет**, без меток P0/P1.

#### Концовка ответа: что НЕ писать vs что писать

Поведенческая ловушка (3-й+ повтор паттерна 28.04, теперь под Stop hook): спрашивать «ок?» вместо действия. Few-shot для in-context learning:

| ❌ Так НЕ надо (default = жду) | ✅ Так надо (default = действую) |
|---|---|
| «Действую сверху вниз. Скажи "ок" если не согласен.» | «Действую сверху вниз. <первый пункт уже выполнен — путь файла + что изменено>.» |
| «Жду «ок» / «ок 1,3» / «стоп N».» | «Сделал п.1, п.2 — артефакты ниже. По п.3 нужен ОК (destructive: deploy на прод).» |
| «Готов сделать сейчас. Если ок — делаю.» | «Сделал. Артефакт: `path/to/file.md`. Следующее — п.2 в плане.» |
| «Делать?» / «Делать что могу?» / «Как смотришь?» | <молча выполнить и показать diff/output> |
| «Нужен ОК на пп.1-3.» (без указания что именно destructive) | «Нужен ОК на п.1 (deploy на прод), пп.2-3 уже выполнил — артефакты ниже.» |
| «Жду подтверждения и продолжаю.» | <Edit/Write/Bash прямо сейчас, потом отчёт что сделано> |

Правило: «жду ОК» легитимно **только** на destructive (deploy, миграция БД, удаление прод-данных, git push --force, ssh с явно destructive командой) — и **только** с явным указанием что именно destructive и почему. Все остальные пункты — выполняю и показываю результат.

Формат пункта — **3 строки**, нумерация без чекбоксов `[ ]`:
- Строка 1: `N. <Что сделать — глагол + объект + путь>. <Время>, <режим: делаю сразу / после ОК / отдельной сессией>.`
- Строка 2: `Зачем: <причина одной фразой>.`
- Строка 3: `Без <фикса/бэкфилла/...> <конкретный риск с временным горизонтом>.`

Шаблон:

```
## План действий

1. Закоммитить и задеплоить hh_agent.py:42 — убран default socks5 proxy. ~2 мин, делаю сразу.
   Зачем: токен HH привязан к IP сервера, через прокси HH видит чужой IP и отзывает.
   Без фикса следующий deploy синкнет старый .env — токен отвалится за 1-2 дня.

2. Backfill hh_chat_id в hr_candidates для 22 кандидатов из shortlist. ~30 мин (Codex), сегодня после ОК.
   Зачем: chat_id есть только в текстовом файле, в БД у всех 632 поле NULL.
   Без бэкфилла следующая выгрузка shortlist опять с placeholder'ами вместо ссылок.

3. Спавн сессии на watchdog инфры (pm2 jsonl → алерт в Telegram). 3-4 ч Codex+Sonnet, отдельной сессией не сегодня.
   Зачем: 3-й раз в 2026 инфра падает молча (VPN 03.04, nginx app.astroguru, сегодня HR).
   Без watchdog 4-й инцидент через месяц-два, опять узнаем случайно.

Действую сверху вниз. «Стоп N» если не согласен с пунктом или приоритетом.
```

**Правила блока:**
- Максимум 5 пунктов. Больше — приоритизируй жёстче, остальное в стратегический долг.
- Ровно 3 строки на пункт. Не помещается — формулировка размазана, режь.
- Не получается сформулировать «зачем» или риск — пункт не нужен.
- Destructive (deploy, миграция, прод-скрипт) — всегда сюда с явной пометкой «после ОК».
- Default action — **«делаю»**, не «жду». Алекс тормозит, не запускает.
- Если действий нет — «План действий пуст, всё закрыто в EXECUTE».

**Анти-формат:**
- ❌ `1. Деплой HR-бота` — стаб, ноль информации
- ❌ Простыня на 6+ строк с полями «Как / Что / Если не сделать / Время» отдельно — лишний шум
- ❌ Метки P0/P1/P2 / 🔴🟡🟢 — порядок и так передаёт приоритет
- ❌ `[ ]` чекбоксы — Алекс их не нажимает, отвечает «стоп N»

## Промежуточный режим (`/retro mid`)

Используй посреди сессии когда:
- 3-й retry / «ещё один фикс»
- Таймбокс 30 мин кончился без результата
- Перед большим коммитом / деплоем — точно root cause, не симптом?
- Перед делегированием Codex — полный ли промпт?
- Сессия > 1 часа, «потерял нить»

4 раздела, без EXECUTE/SCHEDULE (артефактов ещё нет):

1. **ЧТО ИДЁТ СЕЙЧАС** — одна строка где находишься.
2. **ЛОВУШКИ (уже попал?)** — те же что в DETECT, но в режиме «уже сделал», а не «сделаешь ли».
3. **ЧТО УПУСТИЛ** — 2-3 пункта: какой skill/правило не открыл (Decision Router в MEMORY.md), какую проверку не сделал (web_search / git log / чтение кода), какой root cause не проверил.
4. **СМЕНА КУРСА** — одна строка: продолжать / pivot к <конкретно> / стоп и спросить Алекса.

### Pre-merge self-grill (триггер: задача затронула >3 файла, БД, прод-конфиг или деплой)

Перед коммитом серьёзных изменений и до выдачи DoD-артефактов прогнать самопроверку. Не «всё ок» — а явно ответить:

- **5 мест регрессии** — где конкретно может сломаться, и почему именно там (имя файла/функции, не общее «возможно фронт»).
- **3 edge case** — которые я скорее всего не покрыл (пустой ввод, race, RU/EN, timezone, NULL, права).
- **2 архитектурных риска под масштаб** — что выдержит x10 нагрузки/данных, а что начнёт деградировать.
- **3 вопроса production-readiness** — на которые должен быть конкретный ответ артефактом, не «вроде да».

Если хоть один пункт пуст или ответ «не знаю» — стоп, не PR. Дописываю проверку или явно пишу Алексу «вижу риск X, нет верификации, нужен ОК».

## Self-check перед отправкой отчёта (молча)

- Каждая пойманная ловушка имеет действие в EXECUTE или строку в плане?
- По каждому уроку проверена **история повторений**, выбран **самый сильный** механизм из лестницы?
- Если урок 3-й+ раз — feedback запрещён, реально hook/тест/код?
- Все артефакты — с полными путями (`memory/...`, `~/.claude/settings.json`, `<file>:<line>`)?
- План действий ≤ 5 пунктов, каждый ровно 3 строки, без чекбоксов и меток приоритета?
- Destructive действия — в плане с пометкой «после ОК», не выполнены автоматом?
- Объём 30-50 строк? Больше — режу описания, не пункты.
- Шаблон decision-файла не вывалился в чат?

## Принципы

- Без подлиз. Облажался — скажи прямо.
- Факт, оценка, действие. Без «возможно».
- Действие > описание. Можешь сделать — делай. Не можешь — конкретный пункт в плане «делаю после ОК».
- **Root cause > пластырь.** Симптомный фикс без корня = либо докопать и записать в decision, либо явно пометить «осознанный пластырь, root cause откладываем по причине X».
- **Механизм применения > правило в памяти.** Феdback — последняя инстанция; если знание уже было — эскалация по лестнице.

## Антипаттерны

| Не делай | Почему |
|---|---|
| Создать `feedback_X.md` для повторного косяка | Декларативно уже не сработало, нужен hook/тест |
| Патчить поверх плохого фикса после «переделай» | Неудачная попытка остаётся в контексте и тянет качество вниз. Правильно: `/rewind` (Esc+Esc) + попросить себя «summarize handoff: что узнал, что не сработало, что не пробовать» → новая попытка с этой выжимкой как единственным контекстом |
| Выдавать шаблон markdown в чат «скопируй в файл» | Алекс — промежуточное звено, Write сам |
| «Что упустил» как описание без действия | Это и есть «справка вместо действия» |
| Расширять план до 10 пунктов | Без приоритизации = шум, выбери 3-5 |
| Включать стратегический блок без триггеров | Размывает ставки |
| Хвалить ради баланса | «Лучшее» = одна строка |
| Создать decision без `mcp__scheduled-tasks__create_scheduled_task` | Петля разомкнута — архив |
| Метки P0/P1/P2 в плане | Порядок и так передаёт приоритет, метки = косметика |
| Чекбоксы `[ ]` в плане | Алекс не нажимает, формат не для интерфейса |
