# Brainstorming — Design Before Code

## Правило #1
**НЕ пиши код, не создавай файлы, не запускай /spec пока дизайн не утверждён.**

"Простые" задачи — именно там непроверенные допущения тратят больше всего времени.

## Процесс (5 шагов)

### Step 1: Explore Context
```
READ: project files, docs, recent commits (git log -10)
UNDERSTAND: текущая архитектура, constraints, что уже есть
```

### Step 2: Clarifying Questions
```
RULES:
- Один вопрос за раз (не пулемёт)
- Каждый вопрос = конкретная ситуация + рекомендация + "подтверди или поправь"
- НЕ абстрактные варианты (A/B/C) — описывай реальный сценарий простым языком
- Focus: цель, ограничения, критерии успеха
- НЕ спрашивай то, что можно найти в коде
```

Примеры хороших вопросов:
- "Сейчас Анна не может добавить куратора без курса, потому что курс — обязательное поле. Рекомендую сделать курс необязательным для кураторов/админов. Ок?"
- "Таблица показывает 4758 строк, но реальных людей ~900. Рекомендую группировать по человеку, курсы — тегами внутри строки. Ок?"

Примеры ПЛОХИХ вопросов (НЕ делай так):
- "Цель: A) массовый онбординг B) единый реестр людей C) оба" — абстрактно, непонятно к чему
- "Это для текущих пользователей или для привлечения новых?" — можно понять из контекста

### Step 3: Propose 2-3 Approaches
```
FOR EACH approach:
  - Суть в 1-2 предложения
  - Trade-offs (плюсы/минусы)
  - Effort estimate (S/M/L)
  - Risk assessment

LEAD with recommended option + почему
```

#### Приоритизация подходов (при >2 вариантов)

Когда подходов много или нужно аргументировать выбор — используй фреймворк:

| Фреймворк | Формула | Когда |
|-----------|---------|-------|
| **ICE** | Impact × Confidence × Ease (1-10) | Быстрая оценка, дефолт |
| **RICE** | (Reach × Impact × Confidence) / Effort | Когда важен масштаб охвата |
| **Opportunity Score** | Importance × (1 − Satisfaction) | Когда есть данные от пользователей |

**ICE** — дефолт для brainstorm-сессий:
- **Impact** = насколько двигает к цели (1-10)
- **Confidence** = насколько уверены в оценке (1-10)
- **Ease** = насколько просто реализовать (1-10)

Не нужна таблица на 3 подхода. Используй когда вариантов 4+ или решение неочевидно.

### Step 4: Present Design
```
SCALE to complexity:
  - Simple task → 3-5 пунктов
  - Medium feature → architecture + data flow + error handling
  - Complex system → full design doc

ASK approval ПОСЛЕ каждой секции, не в конце
```

Секции (масштабируй по сложности):
1. **Architecture** — компоненты, связи
2. **Data Flow** — откуда данные, куда идут
3. **API Contract** — endpoints, request/response shape
4. **Error Handling** — что может пойти не так
5. **Migration** — если меняется существующее

### Step 5: Transition
```
IF approved:
  SAVE design to handoff-prompt или SPEC-файл
  INVOKE /spec для реализации

IF needs changes:
  ITERATE steps 2-4
```

## Anti-patterns
- Сразу писать код "а потом разберёмся"
- Задать 5 вопросов одним сообщением
- Предложить 1 подход без альтернатив
- Over-design: 200 строк для кнопки
- YAGNI violation: "а ещё добавим на будущее..."

## Когда НЕ нужен brainstorm
- Багфикс (используй /debug)
- Точная спека уже есть (сразу /spec)
- Однострочное изменение
- Рефакторинг без изменения поведения
