# Промпт-инжиниринг

## Когда использовать
- Создание/улучшение промпта (system, user, skill, agent)
- Диагностика "LLM делает не то что нужно"

## Workflow: намерение → принципы → сборка → валидация

---

## 1. Намерение

Заполни ДО написания. Пропуск = неточный результат.

| Поле | Вопрос |
|------|--------|
| **Task** | Что именно должна сделать модель? |
| **Input** | Какие данные получает? |
| **Output** | Формат, объём, структура выхода? |
| **Success** | Как выглядит идеальный результат? |
| **Audience** | Кто потребитель ответа? |
| **Constraints** | Обязательные ограничения? |
| **Examples** | 3-5 пар вход→выход? |

---

## 2. Принципы

### Мотивация рядом с правилом
```
СЛАБО:  "Пиши без многоточий"
СИЛЬНО: "Ответ озвучит TTS — пиши без многоточий и спецсимволов"
```
LLM обобщает "зачем" и применяет шире одного правила.

### Позитивные формулировки вместо запретов
```
"Не используй markdown"  →  "Пиши плавной прозой"
"Не извиняйся"           →  "Отвечай уверенно и прямо"
"Не галлюцинируй"        →  "Цитируй только из предоставленных данных"
"Не пиши длинно"         →  "Ответ: 2-3 предложения"
```
"Не делай X" активирует X через прайминг — 87.5% нарушений запретов (Rana, 2026). Если запрет критичен: запрет + позитивная замена вместе.

### Instruction Sandwich
```
НАЧАЛО   →  цель + критичные ограничения
СЕРЕДИНА →  контекст, данные, примеры
КОНЕЦ    →  повтор только 1-2 hard constraints
```
Порядок блоков меняет accuracy на 30%+. Середина — слабая зона внимания.

### Стиль промпта = стиль ответа
Промпт в markdown → ответ в markdown. Промпт прозой → ответ прозой. Пиши промпт в стиле ожидаемого ответа.

### Спокойный тон
"CRITICAL: You MUST ALWAYS..." → overtriggering. Пиши нормальным языком.

## 3. Сборка

```xml
<role>Кто, специализация — 1 предложение</role>

<context>Данные, документы. Для длинных: данные ПЕРЕД инструкциями</context>

<task>Что сделать — 2-3 шага. Глагол + объект + критерий</task>

<rules>
  КРИТИЧНО: [правило] — потому что [мотивация]
  ВАЖНО: [правило] — потому что [мотивация]
  IF условие → действие
</rules>

<output>Точный формат: JSON Schema / таблица / проза. Enum где возможно</output>

<examples>
  <example>
    <input>...</input>
    <output>...</output>
  </example>
  3-5 штук, разнообразных, с edge cases
</examples>

<self_check>Перед ответом проверь: [критерии из карточки намерения]</self_check>
```

### Техники с доказанным эффектом

| Техника | Когда | Эффект |
|---------|-------|--------|
| XML-теги | Всегда | +15-20% vs plain text |
| Разделители (`---`, `===`) | 2+ секций | +16-24% |
| Таблицы | 5+ однородных элементов | +40% |
| Псевдокод IF/ELSE | Условная логика | +36%, -87% токенов |
| Few-shot примеры | Всегда | Самый надёжный способ управлять форматом |
| Self-check блок | Проверяемые критерии | -50% ошибок |
| Ground in quotes | Длинный документ | "Сначала процитируй, потом отвечай" |

## 4. Валидация

```
□ ЯСНОСТЬ      Понятен человеку без контекста задачи?
□ МОТИВАЦИЯ    Рядом с правилом есть "потому что"?
□ ПОЗИТИВ      Запреты → "делай Y"?
□ ПРИМЕРЫ      3-5 разнообразных вход→выход?
□ ПОРЯДОК      Цель вначале, повтор критичного в конце?
□ КОНФЛИКТЫ    Правила не противоречат друг другу?
□ СТИЛЬ        Промпт в стиле ожидаемого ответа?
□ ТОН          Без MUST/ALWAYS/CRITICAL без нужды?
□ SELF-CHECK   Блок проверки с критериями из карточки?
```

---

## 5. Диагностика "делает не то"

```
СИМПТОМ                     → ДЕЙСТВИЕ
Игнорирует правило          → Перемести в начало или конец промпта
Делает запрещённое          → Переформулируй позитивно
Слишком длинный ответ       → Добавь точный формат в <output>
Галлюцинирует               → Добавь данные + "цитируй только из них"
Непредсказуемый формат      → Добавь 3 few-shot примера
Делает лишнее               → Ослабь тон инструкций
Путает контексты            → Оберни блоки в XML-теги
Противоречивое поведение    → Найди и устрани конфликт правил
```
