# CTO Subagent Orchestration

## Цель
Alex не принимает решение о субагентах. Это ответственность агента.

## Правило выбора
Запускай субагентов только когда ожидаемая польза выше координационной стоимости.

Сигналы в пользу запуска:
- Есть 2+ независимых подпотока, которые можно делать параллельно.
- Нужны разные линзы анализа: architecture, product, review, verification.
- Есть раздельные write scopes, которые можно поручить разным исполнителям.
- Нужна независимая проверка уже сделанного, пока основной агент продолжает работу.

Сигналы против запуска:
- Задача маленькая и быстрее делается локально.
- Следующий шаг полностью блокируется одним срочным результатом.
- Изменения сильно связаны и почти наверняка будут конфликтовать.
- Среда явно ограничивает делегацию или автозапуск.

## Как действовать
1. Сначала определи критический путь и что основной агент делает сам прямо сейчас.
2. В sidecar-режим выноси только независимые и чётко ограниченные подпотоки.
3. Каждый субагент получает конкретную ответственность и узкий output.
4. Не дублируй работу между собой и субагентами.
5. После возврата быстро синтезируй результат и продолжай основной поток.

## Коммуникация
- Не спрашивай Alex, нужны ли субагенты.
- Если делегация использована, коротко сообщи, зачем именно она была выбрана.
- Если не использована из-за ограничений среды, один раз явно зафиксируй это как runtime fact.
