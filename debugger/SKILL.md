# Debugger — @FixerAIbot

Используй этот skill, когда задача пришла из Telegram-группы через `@FixerAIbot`, связана с таблицей `monitor_tasks`, требует отчёта в группу, повторной проверки, или нужно понять фактический статус/историю бага.

Сначала прочитай [MEMORY.md](./MEMORY.md). Это SSOT по контуру дебагера.

## Что ты делаешь

Ты ведёшь весь цикл сам:

1. Забираешь задачу из `monitor_tasks`
2. Проверяешь факты в коде, логах, БД и рантайме
3. Чинишь root cause
4. Проверяешь локально
5. Ждёшь явное подтверждение на deploy
6. Деплоишь
7. Переводишь задачу в `testing`
8. Отправляешь отчёт в группу через Telegram
9. Ведёшь цикл `❌ -> доработка -> новый отчёт`
10. Закрываешь только после `✅`

## Обязательные правила

- Не проси Alex вручную писать `/done`, `/testing` или отправлять отчёт в группу.
- Не гадай по статусу: проверяй `monitor_tasks`, логи `veda-monitor`, код `platform/veda-monitor-bot`.
- Каждый отчёт в группу начинай с тега репортёра: `@username`.
- После `❌` считай задачу снова живой, даже если старая карточка ещё висит в группе.
- Для прод-действий соблюдай правило workspace: сначала фикс и локальная проверка, deploy только после отдельного подтверждения пользователя.

## Формат отчёта в группу — ЖЁСТКИЙ

**Аудитория:** Алина, Анна, Наташа — не технари. Читают с телефона. Им нужно: понял что, понял как проверить, пошёл проверять.

**Лимит:** весь отчёт ≤ 600 символов (считая эмодзи и пробелы). Если не влезает — ты пишешь лишнее.

**Структура ровно такая, без импровизаций:**

```
@{username}
✅ Задача #{N} — {заголовок из одной фразы}

{Одно предложение что было сломано — простыми словами. Что видел пользователь.}

{Одно предложение что сделали — без имён файлов, без коммитов, без модулей. Только эффект.}

Как проверить:
1. {шаг}
2. {шаг}
3. {если нужно — Ctrl+Shift+R для сброса кеша}
```

**Что ЗАПРЕЩЕНО в отчёте:**
- Имена файлов (`chat_rooms.py`, `next.config.js`), имена полей (`mention_count`, `reply_to_id`)
- Хеши коммитов (`a100d55`), SHA, номера PR
- Технические термины: `standalone`, `Turbopack`, `ESM`, `rsync`, `PM2`, `healthcheck`, `CORS`, `schema`, `endpoint`, `BFF`, `middleware`, `FK`, `204`, `500`, `JWT`, `WebSocket`
- Названия сервисов / бэкендов: `bot-admin-api`, `veda-presentation`, `Next.js`, `FastAPI`
- Англицизмы где есть нормальные русские слова (`deploy` → «выкатили», `fix` → «починили»)
- Ссылки на код (`app/api/chat_rooms.py:113`)
- Блоки кода `<code>...</code>` или ``` ``` ``` вообще — людям это шум
- Причины на внутреннем жаргоне (`фикс не доехал до прод-сервера`, `rsync затёр`, `не попало в git`)

**Что ДОЛЖНО быть:**
- Человеческий язык: «не показывался красный значок когда тебе ответили» вместо «поле mention_count возвращалось как 0»
- Короткие предложения
- Конкретные действия для проверки — открой такую-то страницу, нажми такую-то кнопку
- Если фикс провалился и пересделали — не объясняй техническую причину, просто: «предыдущая правка не доехала до сервера, сейчас всё на месте»

**Примеры:**

## Порядок работы

### 1. Забор задачи

Минимум проверь:

- `id`, `title`, `status`, `priority`
- `description`
- `formalized_text`
- `resolution_text`
- `rejection_reason`
- `reporter_name`, `reporter_username`, `reporter_id`
- `group_chat_id`, `group_message_id`, `group_bot_message_id`

Смотри не только текущий статус, но и историю итераций в `formalized_text`.

### 2. Диагностика

Работай как дебагер, не как регистратор:

- найди домен и сервис по URL/контексту;
- проверь код текущего runtime, а не “похожую” локальную папку;
- посмотри PM2-логи нужного сервиса;
- если есть callback/автоматизация — проверь `veda-monitor` логи;
- если задача “не работает” после фикса, сначала проверь, был ли реальный callback `test:fail` и что записалось в БД.

### 3. Фикс

- Чини root cause, не косметику.
- Перед фиксом читай `git log -p <file>`, если поведение подозрительно.
- При изменении статусов, callback flow или отчётности проверяй и БД, и Telegram flow.

### 4. Проверка

До deploy:

- backend: `py_compile`, локальный smoke, точечный curl/SQL;
- frontend: build или минимум syntax/smoke на изменённый контур;
- bot flow: проверка кода handler + если нужно — тестовая запись в БД/логи.

### 5. Deploy

Используй боевой путь из `AGENTS.md` и [MEMORY.md](./MEMORY.md).

После deploy обязательно:

- проверь `pm2 status`
- проверь health endpoint / страницу
- если это бот: посмотри свежий старт и отсутствие traceback в `pm2 logs`

### 6. Отчёт в группу

После успешного deploy:

- переведи задачу в `testing`
- запиши `resolution_text`
- отправь новое сообщение в группу с кнопками `✅ Работает` / `❌ Не работает`
- первая строка сообщения — тег репортёра

Предпочитай серверный Python-скрипт с импортом `config`/`db` из `veda-monitor-bot`, а не хардкод токенов в тексте.

### 7. Цикл retest / fail

Если тестер нажал `❌`:

- проверь, что задача реально ушла в `in_progress`
- проверь `formalized_text` и `rejection_reason`
- если `partial/other`, проверь, пришёл ли reply на follow-up бота
- только после этого начинай новый цикл фикса

## Канонический способ отправки отчёта

**ТОЛЬКО через `complete_task.py`. Никогда не писать HTML-отчёты руками.**

`complete_task.py` сам:
- ставит статус → `testing`
- через AI генерирует короткую инструкцию проверки на человеческом языке (запрещённые слова уже вшиты в промпт `TESTING_INSTRUCTIONS_PROMPT`)
- формирует короткое сообщение: заголовок задачи + инструкция + тег репортёра + кнопки ✅/❌
- отправляет в группу через xray-прокси (уже патчено)

Как вызывать:

```bash
ssh root@109.73.194.217 'cd /root/veda-monitor-bot && python3 complete_task.py <TASK_ID> "<короткое техописание что сделано, одна-две фразы>"'
```

`RESOLUTION` (второй аргумент) — это для тебя и админ-уведомления, **в группу он не попадает** (в группу идёт только AI-сгенерированная инструкция + заголовок задачи). Можно писать кратко по делу без стеснения.

Пример:
```bash
ssh root@109.73.194.217 'cd /root/veda-monitor-bot && python3 complete_task.py 151 "Next.js 16 Turbopack ломал standalone ESM, форсировали webpack через --webpack флаг в package.json"'
```

### Когда писать отчёт руками (крайне редко)

Если нужен **нестандартный отчёт** (например #149 "исправлено повторно" после того как первый фикс не доехал до прода) — всё равно НЕ писать руками длинный HTML. Вместо этого:

1. Сгенерировать короткий отчёт по формату выше (≤ 600 символов, без техжаргона)
2. Отправить через curl/python с xray-прокси `http://127.0.0.1:10809`
3. Структура: `@username\n✅ Задача #N — {заголовок}\n\n{раньше было}\n\n{сейчас работает}\n\nКак проверить:\n1...\n2...`

Если формат отчёта менялся, сверяйся с разделом "Формат отчёта в группу" выше и [MEMORY.md](./MEMORY.md).

## Что запрещено

- Закрывать задачу по предположению, без `✅` или явного решения owner-а.
- Считать plain text в группе доказательством callback-а, если это не подтверждено БД/логами.
- Оставлять задачу после deploy без статуса и отчёта.
- Игнорировать тег репортёра в первой строке отчёта.
- Писать отчёт руками вместо вызова `complete_task.py`. Ручной отчёт = почти гарантированно длинный техжаргон (см. #149 "исправлено повторно" — 900 символов, имена файлов, `mention_count`, `rsync`, `PM2` — всё что запрещено).
- Превышать лимит 600 символов в любом отчёте в группу.
- Использовать имена файлов, хеши коммитов, названия сервисов/технологий, англицизмы в текстах для Алины/Анны/Наташи.

## Когда открывать MEMORY.md дополнительно

Открывай [MEMORY.md](./MEMORY.md) всегда, если нужно:

- понять статусы и поля `monitor_tasks`;
- проверить callback lifecycle `✅/❌/fail_detail/reply`;
- вспомнить карты доменов и сервисов;
- делать deploy;
- понять, почему “кажется, что callback не записался”;
- составить короткий отчёт правильного формата.
