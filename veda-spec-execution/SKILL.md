# Спеки — Lifecycle (подготовка → handoff → исполнение)

## CRITICAL: Выходной формат

```
IF готовишь спеку (исследование → написание SPEC-файла):
  1. Напиши SPEC-*.md файл (БЕЗ handoff-промпта внутри файла)
  2. ВЫВЕДИ handoff-промпт в чат как блок кода (```)
  → Пользователь копирует блок кода в новую сессию
  ⚠️ ЗАПРЕЩЕНО: класть handoff-промпт ВНУТРЬ спеки. Спека = документ. Промпт = в чат.

IF исполняешь спеку (получил промпт → реализация):
  1. Прочитай спеку ЦЕЛИКОМ
  2. Пройди чеклист "ДО начала"
  3. Реализуй по порядку
  4. После КАЖДОГО этапа: build + отчёт в отчетном формате
  → Если не влезает: остановись + handoff-промпт для следующей сессии

IF завершаешь сессию (незаконченная работа):
  1. Сохрани прогресс в SPEC-файле (секция "Что сделано")
  2. ВЫВЕДИ handoff-промпт в чат как блок кода
```

**НЕ пересказывай содержание спеки в чате. ТОЛЬКО блок кода для копипаста.**

Важно: этот skill задает отчетный формат только для execution/handoff/reporting. Он не должен превращать весь диалог с пользователем в шаблонный статус-репорт.

Если пользователь меняет сам шаблон отчёта, handoff или правила спеки, это уже не execution, а `policy editing`: сначала обнови SSOT, потом используй новый шаблон.

### Формат «Отчёт» (CRITICAL — три ветки, не текст)

Пункт 3 отчёта — ВСЕГДА одна из трёх веток, не вольный текст:

```
ВЕТКА A — остались задачи (следующая сессия):
  3. Следующий шаг: [следующая сессия]
  → Сразу вывести handoff-промпт как блок кода (готов к копипасту)

ВЕТКА B — нужно действие пользователя (деплой / миграция / публикация):
  3. Нужно твоё решение: <конкретный вопрос>?
     → да  /  нет
  (Одна строка. Ждать явного ответа. Не действовать без него.)

ВЕТКА C — всё завершено, действий не требуется:
  3. Готово. Задач нет.
```

**Правила выбора ветки:**
- Есть задачи из спеки, которые не выполнены → ВЕТКА A
- Всё выполнено, но требуется деплой / накатка миграции / другое действие на сервере → ВЕТКА B
- Всё выполнено, деплоить не нужно (локальная работа, или деплой уже в плане отдельно) → ВЕТКА C

**Запрещено:** писать "Следующий шаг: сделать X" без немедленного handoff-промпта или вопроса. Текст без действия = бесполезный отчёт.

### Приоритет отчётного формата (критично)

- Для owner-отчётов SSOT по форме — `compact-cto-reporting`.
- Формат: `JTBD` → `Что сделано на этом этапе` → `Следующий шаг [ветка A/B/C]`.
- Менять этот формат можно только после явного запроса пользователя и обновления SSOT.

## Рекомендуемая модель (ПЕРВАЯ строка handoff-промпта)

Рабочий контур — Claude Code. Модели Claude:

| Модель | Когда |
|--------|-------|
| **Opus 4.6** | Архитектурные спеки, сложный дебаг, multi-step рефакторинг, migration-heavy задачи, новые домены. **Дефолт для серьёзных спек.** У Алекса лимит — используй осознанно. |
| **Sonnet 4.6** | Реализация по готовой спеке, CRUD, правки по нескольким файлам, build/fix/verify цикл, типовые фиксы. **Дефолт для execution.** |
| **Composer 2** | Мультиагентное исполнение. Когда спека содержит 3+ независимых этапов, которые можно параллелить. |

**Правило выбора (агент оценивает АВТОМАТИЧЕСКИ, не спрашивает):**
```
IF спека содержит архитектуру + код + миграции + новые слои runtime → Opus 4.6
IF спека — execution по готовому плану, fix/build loop → Sonnet 4.6
IF задача шаблонная, CRUD, UI-правка → Sonnet 4.6
IF сомневаешься → Sonnet 4.6 (экономим Opus)

ОБЯЗАТЕЛЬНО: подставь конкретную модель в промпт. НЕ оставляй {{выбор}}.
```

## Шаблон handoff-промпта

```
Рекомендуемая модель: {{Opus 4.6 | Sonnet 4.6 | Composer 2}}
Задача: {{одна фраза — что меняется для продукта/бизнеса, на понятном языке без техжаргона}}

[ВЫПОЛНИ СПЕКУ ПОД КЛЮЧ]

Спека: ~/Projects/platform/SPEC-{{имя}}.md
Прочитай ЦЕЛИКОМ перед началом.

## Контекст (прочитай если нужно)
- Skill: {{основной скилл}} (методология)
- Skill: {{доп. скиллы}}

## Что уже сделано
{{конкретные файлы и изменения ИЛИ "Ничего, начинаем с нуля"}}

## Ключевые файлы
{{5-10 файлов, которые будут затронуты или являются образцами}}

## Порядок: {{этапы через →}}
## После каждого этапа: npm run build
## После всех: {{финальная проверка из спеки}}

Сдай «Отчет» в формате (нумерованный список, коротко и простым языком):
1. JTBD
2. Что сделано на этом этапе — не больше 34 слов; только суть.
3. Следующий шаг — ТОЛЬКО одна из трёх форм:
   - Остались задачи → написать: "Следующий шаг: [след. сессия]" + сразу вывести handoff-промпт блоком кода
   - Нужно действие пользователя → написать: "Нужно твоё решение: <вопрос>? → да / нет"
   - Всё готово → написать: "Готово. Задач нет."
```

Происхождение лимита для пункта 2: `34 = 50% от примера пользователя в 68 слов`.

## Когда использовать

| Триггер | Режим |
|---------|-------|
| "сделай спеку" / "напиши спеку" / исследование завершено | Подготовка → handoff |
| "выполни спеку" / "реализуй" / "/spec SPEC-файл" | Исполнение |
| сессия заканчивается, работа не завершена | Handoff → следующая сессия |
| "/handoff" | Handoff (тот же процесс) |

## Подготовка спеки

### Обязательные секции SPEC-файла

1. **Заголовок** — проект, статус, приоритет, дата
2. **Контекст** — 3-5 строк что и зачем
3. **Fix/Task #N [SEVERITY]** — пронумерованные, с приоритетом (CRITICAL / HIGH / MED)
4. **Каждый fix/task содержит:**
   - Файлы (путь + строки)
   - Текущий код (дословно, с номерами строк)
   - Решение (конкретный код замены)
   - Критерий готовности
5. **Порядок реализации** — сводная таблица
6. **Проверка после фиксов** — build + тест
7. **Готовый промпт** — заполненный шаблон (секция выше)

### Принцип

Спека настолько детальная, что следующая сессия НЕ тратит контекст на исследование кода. Все файлы, строки, типы, переменные — уже в спеке.

### Pre-mortem (опционально, для M/L спек)

Перед реализацией сложных спек — представь что запуск провалился. Что пошло не так?

Категоризируй риски:
- **Tigers** — реальные проблемы (доказуемые, требуют действий)
- **Paper Tigers** — то, что кажется страшным, но на деле неважно
- **Elephants** — то, о чём никто не говорит, но стоило бы проверить

Для каждого Tiger определи:
- **Launch-blocking** — без решения не начинаем
- **Fast-follow** — решаем в первые 30 дней после
- **Track** — мониторим, фиксим если станет проблемой

```
КОГДА делать pre-mortem:
  - Спека затрагивает >5 файлов
  - Есть миграции БД
  - Затронуты платёжные/auth флоу
  - Первый раз трогаем этот домен

КОГДА НЕ делать:
  - Багфикс
  - UI-правка
  - Seed данные
```

Добавляй секцию "## Риски" в SPEC-файл, если находишь хотя бы один Tiger.

## Исполнение спеки

### Порядок реализации (ВСЕГДА)

```
БД (Alembic) → Модель → API → BFF → TypeScript типы → Фронтенд → build → тесты → деплой
```

### Чеклист ДО начала

- [ ] Спека прочитана ЦЕЛИКОМ
- [ ] Понятно какие файлы затронуты
- [ ] Проверены зависимости (миграции, env-переменные)
- [ ] Связанные скиллы прочитаны

### Чеклист ПОСЛЕ каждого этапа

- [ ] `npm run build` без ошибок
- [ ] Нет broken imports
- [ ] Краткий «Отчет» 1–2–3: JTBD → что сделано (≤34 слов) → действие [handoff-промпт | вопрос да/нет | "Готово"]

### Чеклист ПОСЛЕ всех этапов

- [ ] Чистый билд
- [ ] Проверить страницы из спеки
- [ ] Нет регрессий
- [ ] Осмысленные коммиты по этапам

### Антипаттерны

| Не делай | Делай вместо |
|----------|-------------|
| Пропускать build | Build после каждого этапа |
| Менять файлы без чтения | Read tool перед Edit |
| Всё в один коммит | Один этап = один коммит |
| Забывать BFF | Бэкенд + BFF всегда вместе |
| Реализовывать без команды | Жди "начинай" / "реализуй" |
