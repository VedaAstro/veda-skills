# Compact Reporting (Owner-Friendly)

## Когда использовать

- Нужно сдать отчёт после заметной работы
- Нужен короткий и понятный статус для owner
- Нужно зафиксировать: что сделано и что дальше

## Когда НЕ использовать

- Обычный диалог и короткие ответы
- Уточнения и быстрые проверки
- Ситуации, где формальный отчёт не нужен

## Policy Editing Trigger

Если пользователь меняет сам формат отчёта, лимиты, шаблон или правила этого skill:
- сначала переключись в режим `policy editing`, а не `compact reporting`;
- запрещено подставлять лимиты и формулировки от себя;
- сначала измерь пример/источник пользователя, затем зафиксируй уже готовое число и его происхождение.

## Главный принцип

Пиши так, чтобы отчёт был понятен человеку без техбэкграунда.

Ответ должен читаться за 10–20 секунд.

## Обязательный формат

Сначала всегда отдельной строкой заголовок:

`Отчёт по задаче`

Дальше нумерованный список. Количество пунктов зависит от статуса задачи:

**Если задача ещё не завершена — 4 пункта:**

1. `JTBD`
2. `Что сделано на этом этапе`
3. `Git: commit 🟢/🔴 | push 🟢/🔴 | deploy 🟢/🔴/⚪`
4. `Следующий шаг`

**Если задача готова (задеплоена / передана / закрыта) — 5 пунктов:**

1. `JTBD`
2. `Что сделано на этом этапе`
3. `Git: commit 🟢/🔴 | push 🟢/🔴 | deploy 🟢/🔴/⚪`
4. `Как проверить`
5. `Следующий шаг`

Пункты короткие, по 1–2 предложения.
Пункт `Что сделано на этом этапе` — не больше 34 слов.
Происхождение лимита: `34 = 50% от примера пользователя в 68 слов`.

## Приоритет формата (критично)

- Для owner-отчётов этот skill — SSOT по формату.
- Если в других инструкциях, спеках или промптах встречается другой формат, по умолчанию игнорируй его и используй только этот.
- Исключение: пользователь в текущем сообщении явно требует другой формат. Тогда сначала обнови SSOT через policy editing и только после этого применяй новый шаблон.

## Обязательная прозрачность статуса

В пункте `Что сделано на этом этапе` всегда явно указывать:
- где сделано: `локально` / `в репо` / `на проде`
- что не сделано из этого трека (например: `на прод не выкачено`)

Запрещено оставлять статус неявным.

## Git-статус в отчёте (критично)

Каждый отчёт ОБЯЗАН содержать строку `Git:` после основных пунктов.

Формат:
```
Git: commit 🟢 | push 🟢 | deploy 🔴 (не требуется)
```

Варианты значений:
- `commit 🟢` / `commit 🔴 (N файлов не закоммичено)`
- `push 🟢` / `push 🔴 (N коммитов не запушено)`
- `deploy 🟢 (на проде)` / `deploy 🔴 (не деплоил)` / `deploy ⚪ (не требуется)`

Если commit или push = ✗, это красный флаг. Агент ОБЯЗАН сначала закоммитить и запушить, и только потом выдавать отчёт. Отчёт с ✗ по commit/push допустим только если есть объективная причина (конфликт с remote, чужие файлы).

## Локальные ссылки (критично)

- Нельзя давать локальную ссылку без проверки доступности.
- Перед ответом агент обязан:
  1) проверить, запущен ли локальный сервер;
  2) если не запущен — запустить сам;
  3) проверить ответ по ссылке (например, `HTTP 200/30x`), и только после этого отправлять ссылку пользователю.
- Если сервер не удалось поднять, вместо ссылки дать короткий факт о причине и что уже проверено.

## Язык отчёта (критично)

- Запрещён техжаргон и внутренние термины, если без них можно обойтись
- Разрешённые термины: `фронтенд`, `бэкенд`, `база данных`, `API`
- Писать `репо` вместо `репозиторий`
- Не использовать названия внутренних функций/классов/файлов в отчёте по умолчанию
- Не вставлять англоязычные сокращения и аббревиатуры без необходимости
- Не писать «если хочешь», «могу сделать», когда шаг уже очевиден

## Содержимое пунктов

- `JTBD`: одна фраза про главную задачу всей текущей работы (уровень проекта/спеки), а не про отдельный узкий шаг
- `Что сделано на этом этапе`: только завершённое, простыми словами, без списков, максимум 34 слова
- `Как проверить`: конкретные шаги или URL, по которым можно убедиться что всё работает. Добавляется только когда задача готова.
- `Следующий шаг`: один конкретный шаг
- Если пользователь меняет правило формата, сначала обнови SSOT, потом применяй новый формат.

### Правило фокуса для JTBD (критично)

- JTBD всегда берётся из верхнего контекста задачи: что в целом должно измениться для продукта/бизнес-процесса.
- Запрещено подменять JTBD локальным подциклом (например, только письма, только экран, только API), если это не была исходная цель задачи.
- Если есть сомнение, JTBD формулируется по названию спеки/главному запросу пользователя, а не по последнему выполненному шагу.

Антипример (неправильно): «дать команде контроль писем».  
Правильно в этом контексте: «настроить систему ролей и доступов, где письма — один из обязательных механизмов».

### Правило для пункта "Следующий шаг" (критично)

- Писать `не нужен` можно только если задача полностью закрыта до нужного состояния (включая ожидаемую доставку результата: репо/прод).
- Если есть незавершённая часть (например, нет push/deploy/проверки на проде), следующий шаг обязателен и должен быть конкретным.

## Красные флаги

- Нет заголовка `Отчёт по задаче`
- 4 пункта когда задача ещё не готова (пункт «Как проверить» только при завершённой задаче)
- Много технических деталей и непонятных слов
- Несколько «следующих шагов» вместо одного

## Шаблон

**Задача не завершена:**
```
Отчёт по задаче

1. JTBD: ...
2. Что сделано на этом этапе: ...
3. Следующий шаг: ...
```

**Задача готова:**
```
Отчёт по задаче

1. JTBD: ...
2. Что сделано на этом этапе: ...
3. Как проверить: ...
4. Следующий шаг: ...
```

## Примеры

**Готово:**

Отчёт по задаче
1. JTBD: убрать ручную работу команды при проблемах с письмами.
2. Что сделано на этом этапе: добавлен экран в настройках, где видно статус писем и можно повторить отправку в один клик. На проде.
3. Git: commit 🟢 | push 🟢 | deploy 🟢
4. Как проверить: зайти в sales.myveda.ru → Настройки → Письма — должен появиться новый раздел со статусами.
5. Следующий шаг: не нужен.

**Не готово:**

Отчёт по задаче
1. JTBD: убрать ручную работу команды при проблемах с письмами.
2. Что сделано на этом этапе: добавлен экран, изменения в репо. На прод не выкачено.
3. Git: commit 🟢 | push 🟢 | deploy 🔴
4. Следующий шаг: задеплоить на sales.myveda.ru.

**Красный флаг (так быть НЕ должно):**

Отчёт по задаче
1. JTBD: добавить CTA-кнопку на мобилке.
2. Что сделано на этом этапе: кнопка добавлена, видна на мобилке. Локально.
3. Git: commit 🔴 (3 файла) | push 🔴 | deploy 🔴
4. Следующий шаг: задеплоить.
^ Агент обязан сначала закоммитить и запушить, потом выдавать отчёт.
