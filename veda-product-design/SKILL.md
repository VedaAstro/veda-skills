# Продуктовый дизайн — от принципов восприятия к экрану

> Структура: Фундамент (почему) → Правила вывода (что делать) → Решения (как именно).
> Каждое решение трассируется до фундамента. Если не можешь объяснить ПОЧЕМУ — решение неверное.

---

## СЛОЙ 1: Фундамент (принципы восприятия)

Это не мнения. Это результаты исследований когнитивной психологии и нейрофизики зрения. Они не меняются от проекта к проекту.

### F1. Gestalt-законы группировки

Мозг автоматически группирует визуальные элементы по нескольким принципам:

- **Proximity (близость)**: Элементы рядом воспринимаются как связанные. Элементы далеко — как несвязанные. Это сильнее любых рамок, линий и заголовков.
- **Similarity (подобие)**: Элементы одного формата/цвета/размера воспринимаются как одна группа. Если два блока данных выглядят по-разному — мозг считает их разными категориями.
- **Continuity (непрерывность)**: Глаз следует по линии/направлению. Если элементы выстроены в ряд — мозг читает их как последовательность.
- **Closure (замыкание)**: Мозг достраивает незавершённые формы. Карточка с тенью = отдельный объект, даже без явной границы.
- **Common region (общая область)**: Элементы внутри одного контейнера (карточка, фон, рамка) воспринимаются как группа.

### F2. Pre-attentive processing (до-внимательная обработка)

Некоторые визуальные свойства мозг обрабатывает мгновенно (< 200мс), ДО сознательного внимания:
- **Цвет** — красный среди серых мгновенно заметен
- **Размер** — крупный элемент среди мелких мгновенно заметен
- **Ориентация** — наклонённый элемент среди прямых мгновенно заметен
- **Движение** — анимация в статичном контексте мгновенно привлекает

Числа, текст, смысл — это НЕ pre-attentive. Чтобы понять "1.234 — это хорошо или плохо", мозгу нужно сознательное усилие.

**Следствие**: Для мгновенного сравнения используй визуальные каналы (цвет, размер, длину бара). Для осмысленного анализа — текст и числа.

### F3. Cognitive load theory (Sweller)

Нагрузка на мозг складывается из трёх компонентов:
- **Intrinsic load** — сложность самого контента (астрология сама по себе сложна). Не можем уменьшить.
- **Extraneous load** — нагрузка от плохого дизайна (хаотичная композиция, смешанные форматы, дублирование). МОЖЕМ И ДОЛЖНЫ минимизировать.
- **Germane load** — усилие на построение ментальной модели (обучение). Хотим максимизировать.

**Формула**: intrinsic + extraneous + germane ≤ capacity. Если extraneous load высокий → germane падает → юзер НЕ учится, НЕ понимает, НЕ получает ценности.

Каждый элемент экрана, который не помогает задаче — extraneous load.

### F4. Scan patterns определяются задачей

Не существует "одного правильного" паттерна сканирования. Паттерн зависит от задачи юзера:

| Задача | Паттерн сканирования | Следствие для лейаута |
|---|---|---|
| Чтение текста | F-паттерн (слева→направо, сверху→вниз, снижая длину) | Текст слева, ключевые слова в начале строк |
| Поиск конкретного элемента в списке | Вертикальное сканирование по одной колонке | Искомые элементы — в левой колонке, выровнены |
| Обзор data dashboard | Focal point → контекст. Глаз идёт к визуально доминантному элементу, потом по окружению | Якорь = визуально крупнейший элемент |
| Сравнение значений | Вертикальное сканирование по колонке сравнения | Сравниваемые значения в одном столбце, рядом |

### F5. Visual hierarchy через размер и контраст

Мозг определяет "что важнее" по двум каналам:
- **Размер**: крупнее = важнее (площадь на экране)
- **Контраст**: контрастнее к фону = важнее (чёрный текст на белом > серый на белом)

Если два элемента одного размера и контраста — они конкурируют за внимание. Мозг не знает куда смотреть → extraneous load растёт.

### F6. Chunking (группировка в chunks)

Рабочая память удерживает 3-4 chunk'а одновременно (Cowan, 2001). Chunk = группа элементов, объединённых в один объект (номер телефона "495-123-45-67" = 4 чанка, не 10 цифр).

**Следствие**: На любом экране — не больше 3-4 визуальных групп верхнего уровня. Внутри каждой группы — свои sub-chunks. Если групп > 4 без вложенной иерархии — мозг перегружен.

## СЛОЙ 2: Правила вывода

Каждое правило ниже — следствие одного или нескольких принципов из Слоя 1. Тег `[F1-F6]` показывает происхождение.

### Композиция

**V1. Один визуальный якорь на экран** `[F4: focal point, F5: hierarchy]`
При задаче "обзор dashboard" глаз идёт к визуально доминантному элементу. Если доминантного нет — глаз мечется, extraneous load растёт. Поэтому: один элемент крупнее и контрастнее остальных. Всё остальное — контекст к нему.

**V2. Максимум 3-4 группы верхнего уровня** `[F6: chunking]`
Рабочая память ≤ 4 chunks. Каждая визуальная группа (карточка, секция, блок) = один chunk. Если групп > 4 на одном экране — вводи вложенную иерархию (группы внутри группы) или убирай.

**V3. Proximity определяет связь, не рамки** `[F1: proximity > common region]`
Gestalt proximity сильнее рамок. Два элемента рядом без рамки воспринимаются как более связанные, чем два элемента в одной рамке но далеко друг от друга. Gap между блоками = смысловой разделитель. Не нужны hr, borders, dividers — достаточно пробела.

**V4. Визуальная иерархия через размер и контраст** `[F5: size + contrast]`
Минимум различимых уровней для данного контента. Простой экран (список карт) = 2 уровня. Сложный (chart dashboard) = 3. Количество уровней НЕ фиксировано — оно определяется глубиной контента. Каждый уровень отличается от предыдущего и размером, и контрастом одновременно. Отличие только по размеру или только по цвету — слабая иерархия.

**V5. Всё пространство intentional** `[F3: extraneous load]`
Пустое пространство — это либо breathing room (функция: визуальный разделитель, снижение плотности), либо dead space (нет функции). Dead space = extraneous load. Не конкретный порог в пикселях — а вопрос: "это пространство выполняет функцию? Если нет — убери или заполни."

### Ориентация данных

**V6. Ориентация таблицы = f(задача сканирования)** `[F4: scan patterns]`

Принцип: ось, по которой юзер сканирует чаще всего, должна быть вертикальной (потому что вертикальное сканирование по выровненной колонке = самое быстрое).

| Задача | Что на вертикальной оси | Что на горизонтальной | Результат |
|---|---|---|---|
| "Найти планету X, посмотреть её атрибуты" | Планеты (сущности) | Атрибуты (знак, дом, градус) | Планеты = строки |
| "Какие планеты в доме 7?" | Планеты | Дом (колонка для сравнения) | Планеты = строки, сканируем колонку "дом" |
| "Быстрый обзор 3 ключевых фактов" | — | Факты inline | Key-value inline или compact row |

Для таблицы с N сущностей × M атрибутов, где N > 3 и M > 1 → сущности = строки. Это не абсолют, а следствие: при N > 3 задача "найти конкретную сущность" = основная, а поиск идёт по вертикали.

**V7. Мгновенное сравнение — через визуальный канал** `[F2: pre-attentive]`
Числа = сознательная обработка (медленно). Длина бара, цвет, площадь = pre-attentive (мгновенно). Для задачи "кто сильнее/слабее" — бары/цвет. Для задачи "точное значение" — числа. Часто нужно оба: бар для мгновенного сравнения + число для точности.

**V8. Время — горизонтально** `[F1: continuity]`
Культурно-когнитивная конвенция: время течёт слева→направо (в LTR-культурах). Gestalt continuity: горизонтальная линия читается как последовательность. Timeline, периоды, расписание — горизонтально.

### Содержимое

**V9. Один факт — одно место** `[F3: extraneous load]`
Дублирование = мозг обрабатывает одно и то же дважды + тратит ресурс на проверку "это те же данные или другие?". Каждый факт на экране появляется ровно один раз.

**V10. Числа без контекста = шум** `[F3: extraneous + germane]`
Число "1.234" требует: (1) прочитать, (2) вспомнить шкалу, (3) оценить "хорошо/плохо". Три когнитивных шага. Число "1.2 (выше нормы)" + зелёный цвет: (1) увидел зелёный = хорошо. Один шаг. Контекст (норма, цвет, слово) переводит extraneous load в germane.

**V11. Консистентность формата** `[F1: similarity]`
Gestalt similarity: элементы одного формата воспринимаются как одна группа. Если один блок показывает "Скорпион(Ma), 1 дом", а другой "☽ Рак, 9 бхава" — мозг тратит ресурс на парсинг двух разных форматов. Один формат на экран.

**V12. Progressive disclosure — уровни глубины** `[F3: intrinsic load management]`
Intrinsic load астрологии высок. Нельзя показать всё сразу. Решение: слои.

```
Visible:  Факт              "Луна в Овне, 6 дом"
1 клик:   Значение           "Овен — активные эмоции, импульсивность, 6 дом — здоровье"
Drill:    Полные данные      "15°23', Бхарани, 2 пада, управитель Марс, ЕБ−, ФБ+"
```

Новичок останавливается на 1 клике. Про идёт в drill. Ни один не перегружен.

### Два режима

**V13. Simple и Pro — разные задачи = разные экраны** `[F4: scan patterns, F3: load]`
Задача новичка: "что это значит для меня?" → паттерн = чтение → формат = текст с пояснением, низкая плотность.
Задача про: "все данные быстро" → паттерн = сканирование → формат = таблицы, высокая плотность.
Один лейаут не может оптимально обслуживать оба паттерна. Два режима = два разных экрана, не фильтр.

**V14. В Simple — заменяй, не убирай** `[F3: germane load]`
Убирать данные из Simple mode = снижать germane load (юзер не учится). Заменять данные пояснением = переводить intrinsic load в germane (юзер учится). "Скорпион, 1 дом" → убрать = пустота. "Ваш восходящий знак — Скорпион. Это глубина, интуиция, трансформация." → замена = ценность.

## СЛОЙ 3: Применение (конкретные решения)

> Каждое решение ниже — применение правил из Слоя 2 к конкретному экрану.
> Формат: Решение ← Правило ← Фундамент.

### Chart Overview (dashboard карты)

**Якорь = чарт SVG.**
← V1 (один якорь) ← F4 (focal point при обзоре dashboard) + F5 (самый крупный элемент).
Чарт — визуально доминантный. Всё остальное — контекст к нему.

**Справа от чарта — key info (лагна, солнце, луна, текущий период, йоги).**
← V3 (proximity = связь) ← F1 (Gestalt proximity). Info связана с чартом → рядом.

**Под парой [чарт + info] — таблица планет. Вертикальная: планеты = строки.**
← V6 (ориентация = задача) ← F4 (поиск конкретной планеты = вертикальное сканирование).
10 планет × 3+ атрибутов. Задача: "найти Юпитер → посмотреть дом". Сущности = строки.

**Таблица планет — не больше 5-6 колонок в Pro, 3 в Simple.**
← V2 (≤ 4 chunks) ← F6. Каждая колонка = sub-chunk. Более 6 колонок = горизонтальный скролл или перегрузка.

**Йоги — badges с пояснением (не просто названия).**
← V10 (числа/термины без контекста = шум) ← F3. "Гаджакешари" без пояснения = шум. "Гаджакешари — йога мудрости" = информация.

**Варна — НЕ на overview. Отдельный таб.**
← V2 (≤ 4 групп) ← F6. Overview уже имеет: чарт, info, таблица, йоги = 4. Варна = 5-й → перегрузка.

### Simple mode overview

**Вместо таблицы планет — 3 текстовых блока (лагна, солнце, луна) с описанием.**
← V13 (разные задачи = разные экраны) ← F4 (новичок читает, не сканирует) + V14 (заменяй, не убирай) ← F3 (germane load).

**Текущий период — с человеческим пояснением ("Период Луны — время эмоций и заботы").**
← V10 (контекст) ← F3. "Маха-даша: Луна 2019-2029" — intrinsic load. + пояснение → germane.

### Таблицы данных (Planets tab, Strength tab)

**Шадбала — бары + числа.**
← V7 (мгновенное сравнение через визуальный канал) ← F2. Бар для быстрого "кто сильный", число для точности.

**Аштакаварга — числа в ячейках + цветовой код (зелёный/жёлтый/красный).**
← V7 ← F2. Цвет = pre-attentive канал для мгновенной оценки "хорошо/плохо".

**Даши — горизонтальный timeline.**
← V8 (время горизонтально) ← F1 (continuity).

### Форматирование

**Даты: "15 апреля 1987, 14:30", не "15.04.1987 14:30".**
← V11 (консистентность формата, человеческий) ← F1 (similarity) + F3 (parsing cost).

**Нет ID, timestamps, enum-значений из БД на экране.**
← V11 ← F3. Технический формат = extraneous load.

## Чек-лист

Перед каждым экраном. Каждый пункт трассируется до правила.

```
ВОСПРИЯТИЕ
[ ] Один якорь, визуально доминантный? (V1 ← F4, F5)
[ ] ≤ 4 визуальных групп верхнего уровня? (V2 ← F6)
[ ] Proximity соответствует смыслу? (V3 ← F1)
[ ] Иерархия через размер + контраст, ≥ 2 уровня? (V4 ← F5)
[ ] Нет dead space без функции? (V5 ← F3)

ДАННЫЕ
[ ] Ориентация таблиц соответствует задаче сканирования? (V6 ← F4)
[ ] Сравнение через визуальный канал (бары/цвет), не только числа? (V7 ← F2)
[ ] Нет дублирования фактов между блоками? (V9 ← F3)
[ ] Числа имеют контекст? (V10 ← F3)
[ ] Один формат данных на экран? (V11 ← F1)
[ ] Progressive disclosure: уровни глубины? (V12 ← F3)

РЕЖИМ
[ ] Simple ≠ фильтр Pro? Другая подача, другой экран? (V13 ← F4, F3)
[ ] В Simple: данные заменены пояснением, не убраны? (V14 ← F3)
```

## Как использовать при проектировании нового экрана

```
1. Определи задачу юзера на этом экране (1 предложение)
2. Определи паттерн сканирования для этой задачи (F4)
3. Выбери якорь (V1) — самый крупный элемент, в позиции focal point
4. Группируй контент по смыслу через proximity (V3), ≤ 4 групп (V2)
5. Для каждой группы: определи ориентацию через задачу сканирования (V6)
6. Для чисел: добавь визуальный канал — бар/цвет (V7)
7. Проверь: нет дублирования (V9), формат консистентен (V11)
8. Если два режима: Simple = текст с пояснением (V13, V14), Pro = таблицы с плотностью
9. Пройди чек-лист
```
